Read-only Signals — quick reference

Create a writable signal and expose a read-only view so consumers can read but not mutate state:

// writable inside component
private _count = signal(0);

// expose a read-only view
readonly count = this._count.asReadonly();

// mutate only inside component
this._count.update(c => c + 1); // preferred when new value depends on previous
this._count.set(0);             // replace value (reset)

// template: read the read-only signal using call syntax
// {{ count() }}

When to expose asReadonly()

  • Expose asReadonly() from services or components to prevent external mutation.
  • Keep mutation methods (set/update) encapsulated to enforce validation or side-effects.

Best practices

  • Prefer update(fn) when the new value depends on the previous value.
  • Do not mutate objects/arrays in-place—return new references.
  • Treat asReadonly() as the public read-only API; keep writable signals private.

See Angular Signals docs for advanced patterns (derived signals, effects, stores).
